# Environment variables and secret ownership

## Rules and terminology

This document is an inventory and ownership contract, not an environment file. Every credential-like example is a placeholder. Never copy a real value into Git, a Docker image, a build argument, a workflow input, a support ticket, or command output.

The production configuration is intentionally split by process:

```text
GitHub production environments       deployment-only variables and credentials
Cloudflare Worker configuration      web runtime variable and secret
/opt/careersignals/env/api.env        API-only backend runtime values
/opt/careersignals/env/worker.env     Worker, dbt, MotherDuck, and Connector values
/opt/careersignals/env/scheduler.env  Scheduler and PostgreSQL values only
/opt/careersignals/env/migration.env  migration-only direct database access
release.env                           generated, non-secret immutable release metadata
```

In the matrices below:

- **Public** means the value can appear in browser code. It does not mean the value may be changed by an untrusted user.
- **Server-only** means the value must not appear in browser assets, even when it is not itself a credential.
- **Secret** means the value must be stored by its named secret owner and redacted from logs.
- **Build** means a change requires a new frontend bundle or backend image. **Runtime** means the service reads it while running.
- **Required: gate** means the checked-in production workflow or `verify-environment.sh` refuses to continue without it. **Required: feature** means startup can succeed, but the named feature cannot work safely without it.

Do not place the same secrets in every service. In particular, the API does not receive MotherDuck or Connector credentials, and the Scheduler receives neither Supabase service credentials nor MotherDuck credentials.

## Storage and file controls

| Owner / location | Contents | Required controls |
| --- | --- | --- |
| Canonical GitHub `production` environment | Deployment-repository sync credential | Restricted reviewers; least-privilege secret; required for private-repository clone in both dry-run and push mode; never a workflow input |
| Web-deploy GitHub `production` environment | Cloudflare deployment secret and non-secret build/deploy variables | Manual workflow only; scope token to the account, Worker, and routes; the workflow exposes the token only to Wrangler list/deploy steps |
| Cloudflare Worker | `API_BASE_URL` runtime variable and `PASSWORD_RECOVERY_COOKIE_SECRET` secret | Create the secret once; deployments use `--keep-vars`; do not rotate per release |
| Backend-deploy GitHub `production` environment | Oracle SSH credentials and non-secret host/image metadata | Pinned host key; restricted SSH key; no application data credentials; treat release authority as root-equivalent on Oracle |
| Oracle `/opt/careersignals/env/*.env` | Per-process application values | Regular, non-symlink files; `root:root`; mode `600`; never uploaded or overwritten by deployment |
| Oracle `/opt/careersignals/tls/*` | Origin certificate and private key | Regular, non-symlink files; `root:root`; mode `600` |
| Generated release `release.env` | Image digest and env/TLS directory paths | Non-secret; generated by `install-release.sh`; mode `644` |

Run configuration inspection commands only when their output is redacted. `docker inspect`, `docker compose config`, process environments, and shell tracing can disclose injected secrets.

The deployment verifier enforces the complete production contract below. “Exact” means a different otherwise-valid setting is rejected:

```text
api.env
  CAREERSIGNAL_SAAS_MODE=true
  CAREERSIGNAL_ENVIRONMENT=production
  CAREERSIGNAL_DATA_MODE=postgres
  DATABASE_URL=<non-placeholder>
  SUPABASE_URL=<non-placeholder>
  SUPABASE_SERVICE_ROLE_KEY=<non-placeholder>
  SUPABASE_JWT_AUDIENCE=authenticated
  DEMO_USER_UUID=00000000-0000-4000-8000-000000000020
  DEMO_SESSION_SECRET=<non-placeholder, at least 32 characters>
  CORS_ORIGINS=https://jobs.swiftaihub.com
  LOG_LEVEL=INFO

worker.env
  CAREERSIGNAL_SAAS_MODE=true
  CAREERSIGNAL_ENVIRONMENT=production
  CAREERSIGNAL_DATA_MODE=postgres
  DATABASE_URL=<non-placeholder>
  MOTHERDUCK_TOKEN=<non-placeholder>
  MOTHERDUCK_DATABASE=CareerSignal
  DBT_TARGET=prod
  CAREERSIGNAL_RUN_DBT=true
  CAREERSIGNAL_WRITE_DEBUG_JSON=false
  USER_PIPELINE_MAX_CONCURRENCY=1
  JOB_SOURCES=<all or a comma-separated list of approved real sources>

scheduler.env
  CAREERSIGNAL_SAAS_MODE=true
  CAREERSIGNAL_ENVIRONMENT=production
  DATABASE_URL=<non-placeholder>
  CONNECTOR_REFRESH_CRON=0 7,16,21 * * *
  CONNECTOR_REFRESH_TIMEZONE=America/New_York
  CONNECTOR_REFRESH_TRIGGER_MODE=scheduled
  LOG_LEVEL=INFO
```

Selecting `adzuna`, `serpapi`, `greenhouse`, `lever`, or `usajobs` requires that source's credentials or identifiers described in the Connector matrix; `JOB_SOURCES=all` requires all five sets. The verifier also rejects credentials that must not cross service boundaries and verifies the root-owned TLS files. `CORS_ORIGINS` is an origin and therefore has no `/careersignals` path.

The exact forbidden-key gates are:

```text
api.env       MOTHERDUCK_TOKEN, ADZUNA_APP_KEY, SERPAPI_API_KEY,
              USAJOBS_API_KEY, ADMIN_BOOTSTRAP_PASSWORD
worker.env    SUPABASE_SERVICE_ROLE_KEY, DEMO_SESSION_SECRET,
              ADMIN_BOOTSTRAP_PASSWORD
scheduler.env SUPABASE_SERVICE_ROLE_KEY, MOTHERDUCK_TOKEN,
              DEMO_SESSION_SECRET, SCHEDULER_INTERNAL_SECRET,
              ADZUNA_APP_KEY, SERPAPI_API_KEY, USAJOBS_API_KEY
```

All three env files plus `tls/origin.pem` and `tls/origin.key` must be regular, non-symlink files owned by `root:root` with mode `600`.

The backend image installs the exact transitive versions in `requirements.lock`: the build stage creates a local wheelhouse and the runtime stage installs from it with `--no-index`. Both Python stages use the same digest-pinned `python:3.12-slim-bookworm` image, and `CADDY_IMAGE` uses the digest-pinned multi-platform `caddy:2.11.4-alpine` manifest that includes ARM64. Updating a lock or digest is a reviewed release change and still requires ARM64 build and vulnerability validation.

## Migration-only matrix

`/opt/careersignals/env/migration.env` exists only when the manual backend workflow is allowed to apply migrations.

| Variable | Exposure | Requirement | Meaning |
| --- | --- | --- | --- |
| `SUPABASE_DB_DIRECT_URL` | Secret | **Migration gate** | Complete direct PostgreSQL URI for the exact production project; do not use a transaction-pooler URL. The Supabase CLI receives it as a command argument, so restrict local users and `/proc` visibility during migration |
| `SUPABASE_PROJECT_REF` | Non-secret identifier | **Migration gate** | Project reference; the migration script requires it to appear in the direct database hostname |

The workflow input `backup_reference` is a non-secret marker name, not a connection string or backup credential. The matching regular `root:root` mode-`600` file must already exist at `/opt/careersignals/backups/verified-REPLACE_WITH_REFERENCE`. Because `supabase db push --db-url` necessarily places the direct URL in the CLI process arguments, use a dedicated host with restricted logins and an OS-reviewed `/proc` policy such as `hidepid=2`; verify that policy does not break Docker or host monitoring. Do not run broad process-list diagnostics during the migration, and rotate the database credential if process-argument exposure is suspected.

| `SUPABASE_ANON_KEY` | Parsed but not consumed by current backend behavior | Do not inject it into production services; web uses the publishable key name |
| `SCHEDULER_INTERNAL_SECRET` | Reserved/parsed but currently unused | Do not create or distribute a meaningless secret |
| `CAREERSIGNAL_EXPORT_EXCEL` | No current runtime reader | Remove from production configuration |
| `CAREERSIGNAL_RUN_DBT_TESTS` | No current runtime reader | Remove; fixed dbt builds already run selected tests |
| `CAREERSIGNAL_DBT_FULL_REFRESH` | No current runtime reader | Remove; project-wide full refresh is deliberately disabled |
| `SUPABASE_REGION` | No current application reader | Provider metadata only; do not treat as app configuration |
| `SUPABASE_DB_PASSWORD` | No direct current application reader | Do not expect it to construct `DATABASE_URL`; store the complete URL as one secret |
| `CAREERSIGNAL_LOCAL_DATA_DIR`, `CAREERSIGNAL_OUTPUT_DIR`, `CAREERSIGNAL_EXCEL_PATH` | Local/legacy paths | Not required in PostgreSQL production; do not add writable host mounts for them |
| `CAREERSIGNAL_SKIP_PYTHON_REEXEC` | Deprecated local script control | Not a production service variable |

Configuration aliases are deliberately narrow. Adding a second name for a credential increases rotation and incident risk; update code and tests before introducing any new alias.
